# 09-01: 为什么需要MoE

## Dense模型的算力瓶颈

### GPT系列参数量增长

| 年份 | 模型 | 参数量 | 问题 |
|------|------|--------|------|
| 2019 | GPT-2 | 1.5B | 难以训练 |
| 2020 | GPT-3 | 175B | 算力成本极高 |
| 2023 | GPT-4 | ~1.8T (传闻) | 需要稀疏化 |

### 核心问题

```
GPT-3: 175B参数，但每次前向传播全部175B都要计算
        = 巨大的算力成本
        = 需要超大规模GPU集群
```

### 矛盾点

| 指标 | 需求 | 实际 |
|------|------|------|
| 模型能力 | 参数越多越好 | 需要更多参数 |
| 计算成本 | 希望成本低 | 全量计算太贵 |
| 硬件效率 | 希望高效 | 算力浪费严重 |

## MoE的解决方案

### 核心思想

**不是所有参数都需要参与每次计算**

```
Dense模型:     输入 → [全部参数] → 输出
                          ↓
                     每次100%计算

MoE模型:       输入 → [门控选择] → [部分专家] → 输出
                                      ↓
                               每次只计算5-10%
```

### 类比：医院分诊

```
患者(输入) → 分诊台(门控) → 选择科室 → [专家1/专家2/专家3] → 诊断结果
                              ↑
                        不是所有医生都看
                        只看相关的几个
```

## 参数量 vs 激活参数量

### Dense模型

| 模型 | 总参数量 | 激活参数量 | 激活比例 |
|------|----------|-------------|----------|
| GPT-2 Small | 124M | 124M | 100% |
| GPT-3 | 175B | 175B | 100% |
| LLaMA 7B | 7B | 7B | 100% |

### MoE模型

| 模型 | 总参数量 | 激活参数量 | 激活比例 |
|------|----------|-------------|----------|
| Switch Transformer | 1.6T | 2B | 0.1% |
| Mixtral 8x7B | 46.7B | 12B | 26% |
| **我们的教程模型** | **1T** | **34B** | **3%** |

## MoE的工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                        MoE Layer                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   输入 x                                                    │
│      │                                                     │
│      ↓                                                     │
│   ┌─────────┐                                              │
│   │ 门控    │ → 给每个专家打分                            │
│   │ 网络    │                                              │
│   └─────────┘                                              │
│      │                                                     │
│      ↓                                                     │
│   [0.85, 0.02, 0.01, ...]  (专家分数)                     │
│      │                                                     │
│      ↓                                                     │
│   选择Top-2专家                                            │
│      │                                                     │
│      ↓                                                     │
│   ┌────┐ ┌────┐                                           │
│   │ E1 │ │ E2 │  只调用这2个专家                          │
│   │ 85%│ │ 12%│  按权重加权输出                           │
│   └────┘ └────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## MoE的优势

| 优势 | 说明 |
|------|------|
| **参数量大** | 可以存储更多知识（万亿参数） |
| **算力省** | 每次只激活一部分（10%以下） |
| **效果好** | 专家分工，专业化强 |
| **可扩展** | 增加专家数量不影响推理成本 |

## MoE的挑战

| 挑战 | 描述 | 解决方案 |
|------|------|----------|
| 负载均衡 | 专家被选中频率不均 | Auxiliary Loss |
| 通信开销 | 分布式训练专家在不同GPU | Expert Parallel |
| 显存问题 | 需要同时加载所有专家 | 量化、分片 |
| 训练不稳定 | 路由方差大 | Top-K、噪声 |

## 总结

### 为什么需要MoE？

1. **Dense模型的问题**：算力成本 = 参数量 × 每次前向传播
2. **MoE的解决方案**：稀疏激活，只计算需要的专家
3. **结果**：参数量大幅增加，但计算量基本不变

### 关键概念

- **总参数量**：模型存储所需的参数量
- **激活参数量**：每次前向传播实际计算的参数量
- **稀疏激活**：只选择部分专家参与计算
