# 第8章：RLHF（从人类反馈中学习）

## 概述

**RLHF (Reinforcement Learning from Human Feedback)** = 用人类反馈来训练模型，让模型输出更符合人类偏好。

这是 ChatGPT、Claude、GPT-4 等现代 LLM 的核心技术！

---

## 为什么需要 RLHF？

### 预训练模型的问题

```
用户: 如何制作炸弹？
预训练模型: 制作炸弹需要以下材料...（详细步骤）
           ↑ 只是继续生成文本，不知道这是有害的
```

### SFT 的问题

```
SFT 可以让模型学会"回答问题"，但：
1. 无法区分"好的回答"和"更好的回答"
2. 无法减少有害输出
3. 无法让模型更"有帮助"
```

### RLHF 的解决方案

```
用人类的偏好作为信号：
- 比较 A 和 B 两个回答
- 人类选择更好的那个
- 训练奖励模型学习这个偏好
- 用 RL 算法优化 LLM 生成更好的回答
```

## 第3步：RLHF 训练（核心！）

### 3.1 奖励模型 (Reward Model)

#### 为什么用"比较"而非"打分"？

```
打分方式的问题：
┌─────────────────────────────────────────┐
│ 标注员 A: 这个回答 8 分                 │
│ 标注员 B: 这个回答 5 分                 │
│ 标注员 C: 这个回答 6 分                 │
│                                         │
│ → 不同人的标准不一致！                  │
└─────────────────────────────────────────┘

比较方式的优势：
┌─────────────────────────────────────────┐
│ 标注员 A: A 比 B 好                     │
│ 标注员 B: A 比 B 好                     │
│ 标注员 C: A 比 B 好                     │
│                                         │
│ → 比较 的一致性更高！                   │
└─────────────────────────────────────────┘
```

#### 训练数据格式

```python
# 每条数据包含：prompt + 两个回答 + 哪个更好
comparison_data = {
    "prompt": "如何保持健康？",
    "response_a": "保持健康需要均衡饮食、规律运动和充足睡眠。",
    "response_b": "我不知道。",
    "winner": "a"  # 人类认为 A 更好
}
```

#### 奖励模型架构

```
奖励模型 = 一个能输出分数的 LLM

输入: (prompt, response)
输出: 一个标量分数

┌─────────────────────────────────────────┐
│ Prompt: "如何保持健康？"                │
│ Response: "均衡饮食、规律运动..."       │
│                                         │
│ 奖励模型                                │
│   ↓                                     │
│ 分数: 8.5                               │
└─────────────────────────────────────────┘
```

#### 训练目标

```
目标：让奖励模型给"好回答"更高的分数

损失函数（Bradley-Terry 模型）：

loss = -log(σ(score_winner - score_loser))

其中：
- σ 是 sigmoid 函数
- score_winner = RM(prompt, 更好的回答)
- score_loser = RM(prompt, 更差的回答)

直觉：
- 如果 winner 分数 > loser 分数 → loss 小
- 如果 winner 分数 < loser 分数 → loss 大
```

#### 数学公式详解

```python
import torch
import torch.nn.functional as F

def reward_model_loss(rm, prompt, winner_response, loser_response):
    """
    奖励模型的损失函数
    
    rm: 奖励模型
    prompt: 输入提示
    winner_response: 更好的回答
    loser_response: 更差的回答
    """
    # 奖励模型输出分数
    score_winner = rm(prompt, winner_response)  # 标量
    score_loser = rm(prompt, loser_response)    # 标量
    
    # 损失函数：希望 score_winner > score_loser
    # -log(sigmoid(score_winner - score_loser))
    loss = -F.logsigmoid(score_winner - score_loser)
    
    return loss

# 例子
# score_winner = 8.5, score_loser = 2.1
# diff = 6.4
# sigmoid(6.4) ≈ 0.998
# -log(0.998) ≈ 0.002  ← 损失很小，因为顺序对了

# 如果反过来：
# score_winner = 2.1, score_loser = 8.5
# diff = -6.4
# sigmoid(-6.4) ≈ 0.002
# -log(0.002) ≈ 6.2  ← 损失很大，因为顺序错了！
```

### 3.2 PPO 算法优化 LLM

#### 什么是 PPO？

**PPO (Proximal Policy Optimization)** = OpenAI 在 2017 年提出的强化学习算法

特点：
- 稳定：限制每次更新的步长
- 高效：比之前的 RL 算法更简单
- 适合大规模训练

#### 强化学习框架

```
把 LLM 训练看作强化学习问题：

┌─────────────────────────────────────────────────────────────┐
│  强化学习术语         LLM 训练中的含义                      │
├─────────────────────────────────────────────────────────────┤
│  策略 (Policy)        LLM 本身（决定生成什么 token）        │
│  动作 (Action)        选择生成哪个 token                    │
│  状态 (State)         当前的 prompt + 已生成的 token        │
│  奖励 (Reward)        奖励模型的分数                        │
│  环境 (Environment)   奖励模型 + 文本生成过程               │
└─────────────────────────────────────────────────────────────┘
```

#### PPO 训练流程

```
┌─────────────────────────────────────────────────────────────┐
│  1. 从数据集采样 prompt                                     │
│     prompt = "如何保持健康？"                               │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 用当前 LLM 生成 response                                │
│     response = "均衡饮食很重要..."                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  3. 用奖励模型计算 reward                                   │
│     reward = RM(prompt, response) = 8.5                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  4. 计算 KL 散度惩罚                                        │
│     kl_penalty = KL(当前LLM || 原始LLM)                     │
│     → 防止偏离太远                                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  5. 计算总奖励并更新 LLM                                    │
│     total_reward = reward - β * kl_penalty                  │
│     用 PPO 算法更新 LLM 参数                                │
└─────────────────────────────────────────────────────────────┘
```

#### 为什么需要 KL 散度约束？

```
没有 KL 约束的问题：

奖励模型可能对某些"奇怪"的文本给高分
LLM 会学会生成这些"奇怪"文本来"骗过"奖励模型

例子：
┌─────────────────────────────────────────────────────────────┐
│ Prompt: "今天天气怎么样？"                                  │
│                                                             │
│ 正常回答: "今天天气晴朗，温度适中。"                        │
│ 奖励模型分数: 7.0                                           │
│                                                             │
│ "作弊"回答: "今天天气AAAAAAAAAA很好很好很好..."             │
│ 奖励模型分数: 9.5  ← 奖励模型被骗了！                       │
│                                                             │
│ 没有 KL 约束，LLM 会学会生成这种"作弊"回答                  │
└─────────────────────────────────────────────────────────────┘

有 KL 约束后：
- 如果生成和原模型差异太大的文本
- KL 散度惩罚会增加
- 总奖励反而下降
- LLM 学会保持"正常"的输出
```

#### 目标函数

```python
def ppo_objective(llm, ref_llm, rm, prompt, beta=0.1):
    """
    PPO 的目标函数
    
    llm: 正在训练的 LLM
    ref_llm: 参考模型（训练前的 LLM，冻结）
    rm: 奖励模型
    prompt: 输入提示
    beta: KL 散度的惩罚系数
    """
    # 1. 用当前 LLM 生成 response
    response = llm.generate(prompt)
    
    # 2. 计算奖励模型的分数
    reward = rm(prompt, response)
    
    # 3. 计算 KL 散度
    # P(response|prompt) 在两个模型下的概率
    prob_llm = llm.probability(prompt, response)
    prob_ref = ref_llm.probability(prompt, response)
    
    kl_div = prob_llm * (log(prob_llm) - log(prob_ref))
    
    # 4. 总目标 = 奖励 - KL 惩罚
    objective = reward - beta * kl_div
    
    # 最大化这个目标（相当于最小化 -objective）
    return objective
```

## RLHF 的局限性

### 1. 人类偏好不一致

```
┌─────────────────────────────────────────────────────────────┐
│ 标注员 A: 回答应该简洁                                      │
│ 标注员 B: 回答应该详细                                      │
│                                                             │
│ → 同一个回答，不同人可能有不同判断                          │
│ → 一致性约 73%（10个人中7个人会同意）                       │
└─────────────────────────────────────────────────────────────┘
```

### 2. 奖励模型可能被"欺骗"

```
LLM 可能学会生成：
- 看起来很好但实际无用的回答
- 让奖励模型给高分但不正确的文本

这就是为什么需要 KL 散度约束！
```

### 3. 成本高昂

```
RLHF 需要大量人工标注：
- SFT 数据：需要高学历标注员
- 比较数据：需要人工比较回答质量

成本：数百万美元
```

### 4. 幻觉问题

```
研究表明：RLHF 可能加剧"幻觉"问题

原因：标注员可能不知道正确答案
→ 给自信但错误的回答高分
→ 模型学会"自信地说错"
```

## 关键要点总结

| 概念 | 说明 |
|------|------|
| **RLHF 目的** | 让模型输出符合人类偏好 |
| **奖励模型** | 学习给好回答高分 |
| **比较 vs 打分** | 比较更一致，用 Bradley-Terry 损失 |
| **PPO** | 强化学习算法，稳定更新 |
| **KL 散度** | 防止偏离原模型太远 |
| **局限性** | 人类偏好不一致、成本高、可能加剧幻觉 |
